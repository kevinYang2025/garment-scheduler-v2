# 代码检查计划 — garment-scheduler-v2

## 目标

为项目创建一个系统化的代码检查提示词框架，可用于全面审查代码质量、安全性和可维护性。

## 代码检查提示词模板

```markdown
# 代码审查检查清单 — garment-scheduler-v2

## 审查范围
- [ ] 前端 (garment-web/)
- [ ] 后端 (garment-server/)
- [ ] 配置和文档

## 一、安全性检查 (P0)

### 1.1 认证与授权
- [ ] AUTH_ENABLED 是否在生产环境设为 true
- [ ] API_TOKEN 是否使用强随机值（非默认 garment-dev-token）
- [ ] 是否有 RBAC/角色控制
- [ ] WebSocket 连接是否有认证限制

### 1.2 SQL 注入
- [ ] 所有 SQL 是否使用参数化查询（? 占位符）
- [ ] ORDER BY 子句是否有白名单校验（autoSchedule 的 orderBy）
- [ ] 动态表名/列名是否有安全处理

### 1.3 输入验证
- [ ] 所有 POST/PUT 路由是否有完整的字段校验
- [ ] Excel 导入是否有文件大小/类型限制
- [ ] 数量字段是否校验非负/正数
- [ ] 日期字段是否校验格式

### 1.4 错误信息泄露
- [ ] 生产环境是否暴露 e.message 给客户端
- [ ] 是否返回通用错误消息而非内部细节
- [ ] console.error 是否在生产环境禁用

### 1.5 其他安全
- [ ] CORS 是否限制为特定域名（非 *）
- [ ] 是否添加 helmet 中间件
- [ ] 是否有速率限制（express-rate-limit）
- [ ] 敏感数据（.env）是否在 .gitignore 中

## 二、代码质量检查 (P1)

### 2.1 架构
- [ ] server.js 是否按业务域拆分为路由模块
- [ ] 业务逻辑是否从路由中分离到服务层
- [ ] db.js 是否职责过重（schema + 迁移 + 业务逻辑）
- [ ] 前端是否使用 vue-router 替代手动模块切换
- [ ] 全局状态管理是否引入 Pinia

### 2.2 代码重复
- [ ] fmtLocal() 是否只在一处定义
- [ ] computeFilterOptions() 是否抽取为公共函数
- [ ] hasPermission() 是否实现或移除
- [ ] 批量选择+删除模式是否抽取为 composable
- [ ] 表头滚动同步是否抽取为公共组件

### 2.3 错误处理
- [ ] 空 catch 块是否有注释说明（9处）
- [ ] API 层是否有统一的错误拦截器
- [ ] 前端错误提示是否一致（ElMessage vs console）
- [ ] DELETE 操作是否有级联检查

### 2.4 代码规范
- [ ] 是否引入 ESLint + Prettier
- [ ] 是否有统一的代码格式化配置
- [ ] 是否有 EditorConfig
- [ ] console.log/error 是否在生产环境禁用

## 三、性能检查 (P2)

### 3.1 数据库
- [ ] 列表 API 是否有分页（LIMIT/OFFSET）
- [ ] broadcastSection 是否避免全表查询
- [ ] 是否有 N+1 查询问题（visual-schedule/gantt）
- [ ] 迁移机制是否引入版本号

### 3.2 前端
- [ ] 大文件（600+行）是否拆分为子组件
- [ ] Element Plus 图标是否按需导入（非全量注册）
- [ ] Tailwind CSS 是否实际使用或移除
- [ ] 组件缓存策略是否合理（KeepAlive :max=8）

### 3.3 内存
- [ ] Excel 导入是否有流式处理（非全量加载到内存）
- [ ] 大数据集是否使用 shallowRef
- [ ] 是否有内存泄漏（未清理的事件监听器）

## 四、可维护性检查 (P2)

### 4.1 文档
- [ ] 是否有项目根 README.md
- [ ] 是否有 API 接口文档（OpenAPI/Swagger）
- [ ] 是否有数据库 ER 图
- [ ] 是否有部署文档
- [ ] 是否有变更日志（CHANGELOG）

### 4.2 测试
- [ ] 核心业务逻辑是否有单元测试（排程、库存、交期）
- [ ] API 端点是否有集成测试
- [ ] 关键用户流程是否有 E2E 测试
- [ ] 测试覆盖率是否有目标（如 60%）

### 4.3 部署
- [ ] 是否有 Dockerfile
- [ ] 是否有 docker-compose.yml
- [ ] 是否有 CI/CD 配置（GitHub Actions）
- [ ] 是否有生产环境配置（nginx、PM2）

## 五、业务逻辑检查 (P2)

### 5.1 日期处理
- [ ] 是否修复 toISOString() 违规（ShippingPlan.vue）
- [ ] fmtLocal() 是否统一使用（避免 UTC+8 偏移）
- [ ] 日期格式是否一致（YYYY-MM-DD）

### 5.2 权限系统
- [ ] hasPermission() 是否实现真正的权限检查
- [ ] 是否有角色定义（admin、viewer、editor）
- [ ] 是否有 API 级别的权限控制

### 5.3 数据一致性
- [ ] updateInventory 的 pot_no 是否与 UNIQUE 索引匹配
- [ ] 批量操作是否有事务保护
- [ ] 并发更新是否有乐观锁/悲观锁

## 六、审查输出格式

### 问题分类
- **P0 (阻塞)**: 安全漏洞、数据丢失风险
- **P1 (重要)**: 架构问题、性能瓶颈
- **P2 (改进)**: 代码质量、可维护性
- **P3 (建议)**: 最佳实践、技术债

### 每个问题记录
1. **文件路径**: 具体到行号
2. **问题描述**: 简洁说明
3. **影响范围**: 受影响的功能模块
4. **修复建议**: 具体的改进方案
5. **优先级**: P0-P3

### 审查报告模板
```markdown
# 代码审查报告 — [日期]

## 摘要
- 审查范围: [文件/模块]
- 发现问题: X 个 (P0: X, P1: X, P2: X, P3: X)
- 修复状态: 已修复 X / 待修复 X

## P0 问题 (必须修复)
1. [问题标题]
   - 文件: [path:line]
   - 描述: [问题描述]
   - 影响: [功能影响]
   - 修复: [具体方案]

## P1 问题 (重要)
...

## P2 问题 (改进)
...

## P3 建议 (可选)
...

## 修复进度
- [ ] P0-1: [问题标题]
- [ ] P0-2: [问题标题]
...
```

## 实施步骤

### 阶段一：安全加固 (P0)
1. 修复认证配置（AUTH_ENABLED、API_TOKEN）
2. 修复 SQL 拼接风险（autoSchedule orderBy）
3. 添加 helmet 中间件
4. 统一错误消息（不暴露 e.message）

### 阶段二：架构优化 (P1)
1. 拆分 server.js 为路由模块
2. 引入 ESLint + Prettier
3. 添加输入验证中间件
4. 修复 updateInventory pot_no 问题

### 阶段三：质量提升 (P2)
1. 添加分页支持
2. 抽取公共函数（fmtLocal、computeFilterOptions）
3. 添加单元测试框架
4. 补充项目文档

### 阶段四：部署准备 (P3)
1. 创建 Dockerfile
2. 配置 CI/CD
3. 编写部署文档
4. 添加监控和日志
```

## 验证方式

1. **静态分析**: 运行 ESLint 检查代码规范
2. **安全扫描**: 使用 npm audit 检查依赖漏洞
3. **手动审查**: 按检查清单逐项审查
4. **功能测试**: 验证关键业务流程
5. **性能测试**: 检查大文件导入、并发操作
