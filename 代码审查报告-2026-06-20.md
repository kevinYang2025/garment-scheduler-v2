## 一、严重问题（必须修复）

### S-01: `autoSchedule` 中 `orderBy` 拼接 SQL

- **位置**: `garment-server/db.js:1168`
- **问题**: `autoSchedule()` 函数中将 `orderBy` 变量（来自 `scheduling_strategies.config` JSON）直接拼入 SQL 字符串，虽由后端控制但仍是潜在注入点。
- **修复**: 对 `orderBy` 做白名单校验，只允许 `['due_date ASC', 'plan_qty DESC', 'priority ASC, due_date ASC']` 三个值。

```js
// 当前代码（db.js:1163-1168）
let orderBy = sortField === 'priority' ? 'priority ASC, due_date ASC' :
              sortField === 'plan_qty' ? 'plan_qty DESC' :
              'due_date ASC';
const unplanned = db.prepare(`SELECT * FROM main_plan WHERE is_scheduled = 0 OR is_scheduled IS NULL ORDER BY ${orderBy}`).all();
```

### S-02: 自动排产先清空主计划再重写，无故障保护

- **位置**: `garment-server/server.js:1696-1718`
- **问题**: `POST /api/main-plan/auto-schedule` 直接 `DELETE FROM main_plan` 全表清空再逐条 INSERT。如果写入过程中服务器崩溃，全部主计划数据丢失。
- **修复**: 先写入临时表（或备份到 `main_plan_backup`），确认成功后再替换旧表。或用 `INSERT OR REPLACE` 逐条更新而非全删全写。

```js
// 当前代码（server.js:1696）
const writeTxn = rawDb.transaction(() => {
  rawDb.prepare('DELETE FROM main_plan').run();  // ← 危险：清空后如果后续失败则数据全丢
  // ... INSERT 循环
});
```

### S-03: 每日计划实际产量保存存在竞态条件

- **位置**: `garment-server/server.js:2074-2106`
- **问题**: `POST /api/schedule/sewing-daily-plan/actual` 先 SELECT 查是否存在再决定 UPDATE/INSERT。并发下两请求同时 SELECT 都返回不存在 → 两个 INSERT → 唯一约束冲突。
- **修复**: 改用 `INSERT ... ON CONFLICT DO UPDATE SET ...` 合并为一条原子操作。

```js
// 当前代码（server.js:2082-2100）
const existing = db.get(`SELECT id FROM actual_production WHERE style_no = ? AND ...`);
if (existing) {
  db.run(`UPDATE actual_production SET ... WHERE id = ?`, ...);
} else {
  db.run(`INSERT INTO actual_production ...`, ...);
}
```

### S-04: 目视排程 Move 操作先删后插，失败则丢数据

- **位置**: `garment-server/server.js:4689-4748`
- **问题**: `POST /api/visual-schedule/move` 先 `DELETE FROM schedule_master WHERE id = ?`，再 `INSERT INTO schedule_master ...`。如果插入失败（约束违反等），旧记录已丢失。
- **修复**: 将删除和插入放在同一个 SQLite 事务中，或用 UPDATE 直接改目标字段（workshop/line_team/dates）避免删除。

```js
// 当前代码（server.js:4699 和 4732）
db.run('DELETE FROM schedule_master WHERE id = ?', [scheduleId]);  // 先删
// ... 中间无事务保护
db.run(`INSERT INTO schedule_master ... VALUES (...)`, ...);        // 后插
```

### S-05: 款式 POST 接口用 `id` 字段区分新增/更新，客户端可控

- **位置**: `garment-server/server.js:805-830`
- **问题**: `POST /api/styles` 根据 body 中是否携带 `id` 决定是 CREATE 还是 UPDATE，`id` 完全由客户端控制。客户端可传入任意 `id` 篡改已有数据。`POST /api/main-plan`（:1255）也存在同样问题。
- **修复**: 拆分为 `POST /api/styles`（新增）和 `PUT /api/styles/:id`（更新），与 `PUT /api/main-plan/:id` 风格一致。

```js
// 当前代码（server.js:811）
if (s.id) {
  // 客户端传了 id 就变成更新，有安全隐患
  const existing = db.get('SELECT id FROM styles WHERE id = ?', [s.id]);
  // ...
}
```

### S-06: Session 鉴权和 Token 鉴权互相覆盖，Token 实际不可用

- **位置**: `garment-server/server.js:523-546`
- **问题**: 第一层中间件（`AUTH_ENABLED` 时）检查 token，有 session 则跳过；第二层 `requireAuth` 始终验证 `req.session.user`。token 鉴权通过后 `req.session.user` 仍为空，第二层返回 401，导致 token 完全不可用。
- **修复**: token 鉴权通过后设置 `req.user = { id: null, role: 'api' }`，并让 `requireAuth` 也接受 `req.user`，或者在 token 鉴权通过后创建轻量 session。

```js
// 第一层（server.js:523-535）
if (AUTH_ENABLED) {
  app.use('/api', (req, res, next) => {
    if (req.session && req.session.user) return next();
    const token = req.headers.authorization?.replace('Bearer ', '') || req.query.token;
    if (token !== API_TOKEN) return res.status(401).json({ error: 'Unauthorized' });
    // 这里没有设置 req.user，下面的 requireAuth 仍会返回 401
    next();
  });
}
// 第二层（server.js:545）
return requireAuth(req, res, next);  // req.session.user = null → 401
```

## 二、一般问题（建议修复）

### S-08: 报工日期无"不能大于今天"校验，导致出现未来日期数据

- **位置**: `garment-server/server.js:2938` + `garment-web/src/utils/date.js:3-6`
- **问题**: `POST /api/actual` 只校验了 `production_date` 非空，没有校验不能大于当天。`todayLocal()` 使用客户端 `new Date()` 系统时间，如果客户电脑时钟快了（或用户手动选了未来日期），未来日期的报工数据就会被写入数据库。截图显示出现了 2026-06-21 的数据（今天是 2026-06-20），就是因此产生的。
- **修复**:
  1. 后端 `POST /api/actual` 加校验：
     ```js
     if (r.production_date > fmtLocal(new Date())) {
       return res.status(400).json({ error: '报工日期不能大于今天' });
     }
     ```
  2. `PUT /api/actual/:id`、`POST /api/actual/batch` 同理

```js
// 当前代码（server.js:2936-2938）
if (!r.style_no) return res.status(400).json({ error: '款号不能为空' });
if (!r.production_date) return res.status(400).json({ error: '日期不能为空' });
// ← 缺少"不能大于今天"的校验
```

```js
// 当前代码（garment-web/src/views/CuttingDispatch.vue:32）
production_date: todayLocal(),  // 依赖客户端时间，不准
```

### G-01: `fmtLocal()` 函数在 db.js 和 server.js 重复定义

- **位置**: `db.js:9-14` 和 `server.js:17-22`
- **建议**: 统一在 db.js 中定义并 `module.exports`，server.js 中通过 `db.fmtLocal` 引用。

### G-02: 文件导入方式不一致（base64 vs FormData）

- **位置**: `garment-web/src/api/index.js:43-52`（importStyles 用 base64）vs `:90-93`（importMainPlan 用 FormData）
- **建议**: 统一使用 FormData + multipart 方式，更高效且支持大文件。

### G-03: API 方法名冲突

- **位置**: `garment-web/src/api/index.js:167-168` 和 `:204-205`
- **问题**: `getSystemConfig`/`updateSystemConfig` 被定义了两次，第 204-205 行的定义覆盖了第 167-168 行。第 204 行实际指向 `/system-config`（旧路由），第 167 行指向 `/config/system`（新路由）。
- **建议**: 删除第 204-205 行的重复定义，确认前端是否在使用旧路由。

### G-04: 裁剪排程导出 Excel 后，裁剪排程 API 泄露 `s.cutting_start` 原始值到前端用于排序

- **位置**: `garment-server/server.js:1823-1830`
- **建议**: 排序逻辑中硬编码 `'9999-99-99'` 作为空值兜底，建议使用常量或 nulls last 逻辑。

### G-05: Dashboard 缺少 WebSocket 数据就绪 guard

- **位置**: `garment-web/src/views/Dashboard.vue`
- **问题**: Dashboard 依赖 WebSocket 推送的 `db` prop，但如果 WebSocket 连接延迟，ECharts 可能在数据未就绪时渲染。
- **建议**: 添加 `v-if="db.initialized"` 或 Loading 态。

### G-06: 汇总 API 中 `quality_rate` 可能返回 NULL

- **位置**: `garment-server/server.js:2878`
- **问题**: `completed_qty` 和 `defect_qty` 都为 0 时 `NULLIF` 返回 NULL，`quality_rate` 为 NULL。
- **建议**: 使用 `COALESCE(ROUND(...), 0)` 包装。

### G-07: 生产环境 `secure: true` 但缺少 HTTPS 强制跳转

- **位置**: `garment-server/server.js:109`
- **问题**: Session cookie 的 `secure` 在生产环境为 true，但 Express 没有配置 HTTP→HTTPS 跳转。如果用户通过 HTTP 访问，cookie 不会被发送。
- **建议**: 在 Nginx 层面或 Express 中添加 HTTPS 跳转中间件。

### G-08: 调试日志残留

- **位置**: `garment-server/server.js:2413,2416`
- **问题**: `PUT /api/schedule/:scheduleType/:id` 中两个 `console.log(JSON.stringify(...).slice(0, 200))` 为调试残留。
- **建议**: 生产环境移除或改为 debug 级别日志。
