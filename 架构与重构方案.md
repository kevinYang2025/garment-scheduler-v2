# 架构与重构方案

**项目**: 制衣工厂生产排程系统 (garment-scheduler-v2)
**整合日期**: 2026-06-20
**整合者**: AI Assistant
**基于**: `代码架构.md`(精简版参考架构) + 实际项目代码(garment-web + garment-server)

---

## 一、整合原则

1. **保留实际项目的所有增强功能** — Socket.IO、工作日历、自动排产、目视甘特图、ASN/DN 等不在精简版文档中,但已被生产环境验证
2. **吸收文档的模块化思想** — 后端按业务拆 NestJS 模块,前端按业务拆 API/Views 子目录
3. **保持数据库 SQLite** — 单机部署,数据已积累,WAL 模式并发 OK,无必要迁 MySQL
4. **分阶段迁移** — 不 freeze,新旧双轨并行,模块成后逐个切换
5. **测试驱动** — 复用现有 38 个单元测试,新增 NestJS 版本必须 100% 通过

### 2.5 Dockerfile 规范(新增,2026-06-20 审核补充)

**前端 `garment-web/Dockerfile`** — 多阶段构建,产物为纯静态资源,**严禁** `node_modules`:

```dockerfile
# Stage 1: 构建
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Stage 2: 镜像 (nginx 提供静态服务,不进 node_modules)
FROM nginx:1.27-alpine
COPY --from=builder /app/dist /usr/share/nginx/html
COPY nginx.conf /etc/nginx/conf.d/default.conf
# 严禁: COPY --from=builder /app/node_modules ...
```

**后端 `garment-server-nest/Dockerfile`** — alpine 基础 + **只装 `dependencies`**:

```dockerfile
# Stage 1: 构建 (含 devDeps 用来编译 TS)
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
COPY tsconfig*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Stage 2: 运行时 (仅 prod deps)
FROM node:20-alpine
WORKDIR /app
ENV NODE_ENV=production
COPY package*.json ./
RUN npm ci --omit=dev          # 关键: --omit=dev
COPY --from=builder /app/dist ./dist
EXPOSE 3002
CMD ["node", "dist/main.js"]
```

**镜像体积参考**:
- 前端镜像 `nginx:1.27-alpine` + dist ≈ **50MB**(无 node_modules)
- 后端镜像 `node:20-alpine` + dist + prod deps ≈ **180MB**
- 不优化的反例(直接 `COPY . .`): 前端 600MB+,后端 800MB+

### 3.2 Phase 5/6 并发写测试规范(新增)

**幂等性原则**(2026-06-20 二次审核补充):压测必须避免污染业务逻辑校验。

1. **多目标**:5 个并发请求对应 **5 个不同的 `style_id`**(或不同产线),避免同一款式被重复排产导致产能翻倍
2. **状态隔离**:`beforeEach` 重置 `:memory:` 数据库,保证每个 spec 从零状态开始
3. **读后再写**:并发前先 `SELECT` 该 style 现有排程数,断言前后差值 = 5(每个 style 恰好被排一次)

```js
// test/concurrent/auto-schedule.spec.ts
import { initTestDb, resetDb, seedStyles } from '../helpers/db';

describe('autoSchedule 并发写压测', () => {
  beforeEach(async () => {
    await resetDb();
    await seedStyles(10);  // 准备 10 个测试款式
  });

  test('5 个并发请求,各自排不同 style,数据不冲突', async () => {
    const targetStyleIds = [1, 2, 3, 4, 5];  // 5 个不同款式

    const promises = targetStyleIds.map(style_id =>
      request(app)
        .post('/api/main-plan/auto-schedule')
        .set('Cookie', authCookie)
        .send({ style_id })
    );

    const results = await Promise.all(promises);

    // 断言 1:全部 2xx
    results.forEach(r => expect(r.status).toBeLessThan(400));

    // 断言 2:日志无锁冲突
    expect(captureLogs()).not.toMatch(/database is locked|SQLITE_BUSY/);

    // 断言 3:每个 style 恰好产生一组排程(无重复,无丢失)
    for (const style_id of targetStyleIds) {
      const count = await db
        .prepare('SELECT COUNT(*) AS n FROM main_plan WHERE style_id = ?')
        .get(style_id);
      expect(count.n).toBe(1);
    }
  });

  test('同一 style 10 并发,验证互斥锁或事务隔离(可选)', async () => {
    // 此用例针对已实现乐观锁/SELECT FOR UPDATE 的场景
    // 当前 autoSchedule 用 db.transaction(),期望 10 个并发最多成功 1 个
    const promises = Array(10).fill(0).map(() =>
      request(app).post('/api/main-plan/auto-schedule')
        .set('Cookie', authCookie)
        .send({ style_id: 1 })
    );
    const results = await Promise.all(promises);
    const successCount = results.filter(r => r.status === 200).length;
    expect(successCount).toBeLessThanOrEqual(1);  // 业务幂等性
  });
});
```

### 3.4 DTO 校验与旧接口兼容规范(新增)

NestJS `class-validator` 严格校验可能误杀旧请求。规则:

1. **所有字段默认 `@IsOptional()`** — 不在 DTO 显式定义的字段会被 `ValidationPipe({ whitelist: true })` 静默丢弃(避免 400)
2. **旧逻辑容忍的"非法值"**:
   - 数字字段收到字符串 → 用 `@Type(() => Number)` + `enableImplicitConversion: true`
   - 日期字段收到空串 → `@Transform(({ value }) => value === '' ? undefined : value)`
   - 必填但旧逻辑不传的字段 → `@IsOptional()` 不报错,在 Service 层兜底默认值
3. **每 Phase 完成后跑回归**:
   - 从 3001 抓取最近 100 次成功请求的 payload(开发环境代理截获)
   - 同一份 payload 重放 NestJS,期望全部 2xx
4. **白名单机制**:
   ```ts
   app.useGlobalPipes(new ValidationPipe({
     whitelist: true,           // 丢弃未声明字段(不报错)
     forbidNonWhitelisted: false,
     transform: true,
     transformOptions: { enableImplicitConversion: true },
   }));
   ```

### 6.3 前端动态路由守卫(新增,2026-06-20 审核补充)

为避免"某模块已迁 NestJS、某模块还在 Express"时前端请求错位,前端需按模块动态指向后端地址:

**核心机制** — 模块路由表 `src/config/backend-routing.js`:

```js
// 模块迁移状态由后端 /api/system/migration-status 提供,前端启动时拉取一次并缓存
export const ROUTING_RULES = {
  // value: 'nest' = 走 3002,'express' = 走 3001
  '/api/auth':         'nest',       // Phase 2
  '/api/system':       'nest',       // Phase 3
  '/api/styles':       'nest',       // Phase 4
  '/api/main-plan':    'nest',       // Phase 5
  '/api/schedule':     'nest',       // Phase 5
  '/api/report':       'nest',       // Phase 6
  '/api/warehouse':    'express',    // Phase 7 (Scope Freeze)
  '/api/asn':          'express',
  '/api/dn':           'express',
};
```

**axios 请求拦截器** `src/api/request.js`:

```js
import { ROUTING_RULES } from '@/config/backend-routing';

const BACKENDS = {
  nest:    'http://localhost:3002',
  express: 'http://localhost:3001',
};

http.interceptors.request.use(cfg => {
  const rule = Object.entries(ROUTING_RULES).find(([p]) => cfg.url.startsWith(p));
  cfg.baseURL = BACKENDS[rule?.[1] ?? 'nest'];
  return cfg;
});
```

**路由守卫 `src/router/guards.js`** 双校验:

1. **角色权限** — 原有逻辑(菜单可见性)
2. **后端就绪** — 进入模块页面前 `GET /api/system/migration-status`,若目标模块未迁 NestJS 则弹提示并降级走 Express

### 6.3.1 优雅降级(2026-06-20 二次审核补充)

`/api/system/migration-status` 拉取失败时,前端 **不能卡死**。降级策略:全量请求降回 Express 3001(旧系统完整可用,容错性最高)。

`src/config/backend-routing.js` 完整实现:

```js
import axios from 'axios';

// 模块迁移状态:启动时从后端拉取一次,失败降级全量 Express
const ROUTING_RULES = {
  '/api/auth':         'nest',
  '/api/system':       'nest',
  '/api/styles':       'nest',
  '/api/main-plan':    'nest',
  '/api/schedule':     'nest',
  '/api/report':       'nest',
  '/api/warehouse':    'express',
  '/api/asn':          'express',
  '/api/dn':           'express',
};

const BACKENDS = {
  nest:    'http://localhost:3002',
  express: 'http://localhost:3001',
};

let routingReady = false;
let routingRules = { ...ROUTING_RULES };  // 默认按静态表

// 启动时拉取后端真实状态,失败保持静态表(全量默认走 nest,与方案一致)
export async function initBackendRouting() {
  try {
    const { data } = await axios.get('/api/system/migration-status', {
      timeout: 3000,
      // 注意:此时 axios 拦截器尚未挂载,直接走相对路径
    });
    if (data?.modules) {
      routingRules = { ...ROUTING_RULES, ...data.modules };
    }
    routingReady = true;
  } catch (err) {
    // 降级:保持静态表,不抛错,前端继续可用
    console.warn('[backend-routing] 拉取迁移状态失败,使用静态路由表', err.message);
  }
}

export function resolveBackend(url) {
  const rule = Object.entries(routingRules).find(([p]) => url.startsWith(p));
  return BACKENDS[rule?.[1] ?? 'nest'];
}

export { routingReady };
```

`main.js` 启动顺序(必须 **先降级再挂载应用**,保证首屏请求可用):

```js
import { initBackendRouting } from '@/config/backend-routing';

// 异步初始化,不等结果(降级逻辑保证不会阻塞)
initBackendRouting().finally(() => {
  app.mount('#app');
});
```

**降级语义**:
- ✅ `migration-status` 返回成功 → 按后端真实状态路由
- ⚠️ `migration-status` 超时/404/500 → 静态表兜底(默认全量 nest,与方案一致)
- ⚠️ 用户操作过程中后端挂掉 → 已挂载应用继续走最后一次成功的规则,新请求报错由 axios 拦截器统一处理

**Nginx 兜底**: 上述前端路由只对开发期 `vite dev` 生效。生产环境全部走 Nginx 80 端口,Nginx 配置按路径前缀分流(见 §2.4)。

## 八、风险与缓解

| # | 风险 | 等级 | 缓解 |
|---|------|------|------|
| R1 | TypeORM schema 与 db.js 不匹配 | 高 | Entity 手动写,不加 auto-sync,初始化时 `SHOW TABLES` 验证 |
| **R2** | **Session 跨进程共享(双轨并行期核心风险)** | **🔴 高** | **Phase 1 必须落地共享 store:**<br>1) 优先 Redis (`connect-redis` + `ioredis`);<br>2) 无 Redis 资源时退回 file-store (`session-file-store`);<br>3) cookie `secure: false`(内网 HTTP),`sameSite: 'lax'`,`httpOnly: true`;<br>4) 必须在 Phase 1 完成后跑 e2e: 同一 session 在 3001 登录,请求 3002 必须识别为已登录 |
| R3 | bcrypt/密码 hash 不一致 | 中 | NestJS 用 `bcrypt`(与 Express 同包同版本,直接复用 3001 用户的 hash,无需迁移) |
| **R4** | **SQLite WAL 写并发瓶颈(双进程 + 长事务)** | 中 | 1) TypeORM 配置 `busy_timeout: 5000`;<br>2) Phase 5/6 增加并发写压测(`autoSchedule` 并发 5 个请求,验证不出现 `database is locked`);<br>3) 报工/排产/入库写操作全部走 `db.transaction()`,避免半提交 |
| **R5** | **Socket.IO 跨进程广播(双轨并行期)** | 中 | 1) Phase 8 引入 `@socket.io/redis-adapter`,所有广播走 pub/sub;<br>2) 若 Redis 暂未到位,实时强相关模块(报工/入库)必须 **同日完成切换**,不允许跨进程长共存;<br>3) Gateway 沿用现有 token `timingSafeEqual` 鉴权 |
| R6 | 单元测试在新框架失效 | 中 | 复用 db.js 测试代码,通过 `:memory:` DB 跑 |
| R7 | DTO 校验严格化导致旧请求 400 | 中 | 详见 §3.4 — 旧字段全保留 `optional`,新校验用 `@IsOptional()` 兜底 |
| R8 | 前端 import 路径改动量大 | 低 | 用 IDE refactor + CI i18n 扫描验证 |
| R9 | 工厂生产中切换 | 高 | 每 Phase 后回退预案,周末维护窗口切 |

### 8.1 P0 风险防线(必须落地)

`R2` / `R4` / `R5` 是双轨并行能否成功的 **基石**。Phase 2 之前必须全部完成:

- **Redis 服务**: 单机部署 `redis:7-alpine`(docker-compose 单独 service),`maxmemory 128mb`,`appendonly no`
- **Session store**: `connect-redis@7` + `ioredis@5`,key 前缀 `sess:`,TTL 跟随 cookie maxAge
- **Socket.IO adapter**: `@socket.io/redis-adapter@8`,pub/sub 走同一 Redis 实例
- **TypeORM 写配置**:
  ```ts
  new DataSource({
    type: 'sqlite',
    database: 'data.sqlite',
    synchronize: false,           // 严禁
    busyErrorRetry: 5,            // 写冲突重试
    prepareDatabase: db => {
      db.pragma('journal_mode = WAL');
      db.pragma('busy_timeout = 5000');
      db.pragma('synchronous = NORMAL');
      db.pragma('foreign_keys = ON');
    },
  });
  ```
