# 架构与重构方案

**项目**: 制衣工厂生产排程系统 (garment-scheduler-v2)
**整合日期**: 2026-06-20
**整合者**: AI Assistant
**基于**: `代码架构.md`(精简版参考架构) + 实际项目代码(garment-web + garment-server)

---

## 一、整合原则

1. **保留实际项目的所有增强功能** — Socket.IO、工作日历、自动排产、目视甘特图、ASN/DN 等不在精简版文档中,但已被生产环境验证
2. **吸收文档的模块化思想** — 后端按业务拆 NestJS 模块,前端按业务拆 API/Views 子目录
3. **保持数据库 SQLite** — 单机部署,数据已积累,WAL 模式并发 OK,无必要迁 MySQL
4. **分阶段迁移** — 不 freeze,新旧双轨并行,模块成后逐个切换
5. **测试驱动** — 复用现有 38 个单元测试,新增 NestJS 版本必须 100% 通过

## 三、分阶段迁移计划

### 总览

| Phase | 内容 | 周期 | 风险 | 验收 |
|-------|------|------|------|------|
| 1 | NestJS 骨架 + TypeORM + SQLite | 0.5 天 | 低 | `/health` 返回 200 |
| 2 | common 模块 (auth/filter/utils) | 0.5 天 | 低 | 单元测试通过 |
| 3 | system 模块 (用户/角色/日志) | 1 天 | 中 | 登录 + 改密端到端 OK |
| 4 | base 模块 (款式/产线/车间) | 1 天 | 中 | 款式 CRUD OK |
| 5 | plan 模块 (主计划/排程) | 2 天 | 高 | 自动排产 + 三行模型 OK |
| 6 | report 模块 (5 工序报工) | 2 天 | 高 | 报工 + recalc OK |
| 7 | warehouse 模块 (Scope Freeze) | 1 天 | 低 | 入库/出库/库存 OK |
| 8 | Socket.IO @WebSocketGateway | 0.5 天 | 中 | 实时推送 OK |
| 9 | 前端 API + views 拆分 | 1 天 | 低 | 22 个文件全移 OK |
| 10 | Docker + nginx + compose | 0.5 天 | 低 | 一键启动 OK |
| 11 | Nginx 分流切换 + 旧代码下线 | 1 天 | 中 | 全量切 NestJS,garment-server/ 标记 DEPRECATED |

**总工期**: 10-11 工作日 (2 周)

### 关键原则

1. **每 Phase 完成后必须**:
   - `npm test` 38+ 单元测试全过
   - `npm run scan-i18n` 208/208 i18n 化 100%
   - 对应模块端到端冒烟(手动测)

2. **数据兼容性**:
   - 同一份 `data.sqlite`,NestJS 与 Express 并行读写
   - TypeORM schema 必须匹配 db.js 已建好的表结构(不能自动 sync,会丢数据)
   - 用 `synchronize: false`,只在初始化时验证 schema

3. **接口兼容**:
   - NestJS 端点 URL 必须与 Express 完全一致
   - 请求/响应 JSON 结构不变
   - Socket.IO event 名称不变

4. **回退预案**:
   - 每 Phase 完成后,Nginx 配置可一键切回 Express
   - 旧 `garment-server/` 标记 DEPRECATED 但保留可启动

## 四、业务模块划分

### 4.1 模块边界

```
system (基础)
  ├─ user, role, log, config
  └─ 提供: 鉴权守卫、操作日志 helper、系统参数

base (基础资料)
  ├─ style, style-color-size, fabric, workshop,
  │  production-line, sewing-workshop-tree, capacity
  └─ 依赖: system (用 user.workshop)

plan (计划)
  ├─ main-plan, schedule-master, schedule-daily,
  │  visual-schedule, auto-schedule, strategy,
  │  delivery-estimation
  └─ 依赖: base

report (报工)
  ├─ cutting, cutting-second, printing, embroidery,
  │  template, ironing, sewing, dispatch-summary
  └─ 依赖: plan

warehouse (仓库, Scope Freeze)
  ├─ inbound, outbound, inventory
  └─ 依赖: report (报工生成待入库单)

workflow (工作流)
  ├─ asn, dn
  └─ 独立模块,不依赖其他

gateway (Socket.IO)
  ├─ 监听所有 module 的事件
  └─ 实时推送给前端
```

### 4.2 共享数据库约束

- 所有 module 共用 `data.sqlite`
- TypeORM Entity 用 `@Entity({ name: 'table_name' })` 映射已存在表
- migration 用 TypeORM CLI 生成,但 disable auto-sync
- 测试用 `:memory:` 数据库,每个 spec 独立初始化 schema

## 五、代码风格规范

### 5.1 后端 (NestJS/TypeScript)

- **Controller**: 路由 + 参数校验 + 调 Service,不写业务
- **Service**: 业务逻辑 + DB 读写,不写 HTTP
- **Entity**: 纯 TypeORM 装饰器,不放方法
- **DTO**: 用 class-validator 校验入参
- **命名**:
  - 文件 kebab-case (`user.controller.ts`)
  - 类 PascalCase (`UserController`)
  - 方法 camelCase (`findById`)
- **错误处理**: 抛 `HttpException`,由全局 Filter 统一响应

### 5.2 前端 (Vue 3/TypeScript)

- **API**: 每个模块一个文件,函数命名 `getXxx`/`createXxx`/`updateXxx`/`deleteXxx`
- **Component**: `<script setup lang="ts">`,Composition API
- **Store**: Pinia,按业务拆 store
- **Router**: 嵌套路由 + 懒加载

### 5.3 测试

- **单元测试**: `*.spec.ts` (NestJS) / `*.test.js` (Express)
- **E2E 测试**: `test/e2e/*.e2e-spec.ts` (NestJS + supertest)
- **i18n 扫描**: `npm run scan-i18n` 必须 100%

## 六、迁移期间接口兼容矩阵

### 6.1 同端口并行方案(默认)

- Express 在 3001,NestJS 在 3002
- 前端 `vite.config.js` 配 proxy `/api` → 3001 (迁移期)
- 每完成一个 module,proxy 改 → 3002
- 最终全量切到 3002 后,3001 关闭

### 6.2 同进程并行方案(高级)

- 在 Express `server.js` 里动态 import NestJS module
- 同一进程内挂载 `/api/new/*` 走 NestJS
- 风险: NestJS 依赖多,require 链复杂,不推荐

**决策**: 采用方案 6.1,简单可控。

## 七、Scope Freeze 策略

迁移期间以下模块按 CLAUDE.md 现有 Scope Freeze 处理:

- **warehouse 模块** — 工厂在用,不主动重构,Phase 7 仅做最小迁移
- **garment-scheduler/** (Java) — 永久退役,见 `garment-scheduler/DEPRECATED.md`

迁移完成后,Scope Freeze 模块保留运行状态,不在 v3 重构范围内。

## 八、风险与缓解

| 风险 | 等级 | 缓解 |
|------|------|------|
| TypeORM schema 与 db.js 不匹配 | 高 | Entity 手动写,不加 auto-sync,初始化时 `SHOW TABLES` 验证 |
| bcrypt/session 跨进程共享 | 中 | 共享 express-session store(Redis 或 file store) |
| Socket.IO 鉴权 | 中 | Gateway 沿用现有 token timingSafeEqual |
| 单元测试在新框架失效 | 中 | 复用 db.js 测试代码,通过 `:memory:` DB 跑 |
| 前端 import 路径改动量大 | 低 | 用 IDE refactor + CI i18n 扫描验证 |
| 工厂生产中切换 | 高 | 每 Phase 后回退预案,周末维护窗口切 |
