## 段7 — P3 system_config 内存缓存 (2 项, commit 81c22f6)

**问题**: 热路径每次都 `SELECT config_key, config_value FROM system_config`,浪费 1 次 DB 查询 + 6 行 for 循环重建 dict。

| 热点路径 | 触发频率 |
|---|---|
| `recalcMainPlanDates()` | 每次 POST/PUT `/api/main-plan` |
| `auto-schedule` | 每次 POST `/api/main-plan/auto-schedule` |

**方案**: lazy 加载 + 显式 invalidate:
```js
let _configCache = null;
function getSystemConfig() {
  if (!_configCache) {
    const rows = db.all('SELECT config_key, config_value FROM system_config');
    _configCache = {};
    for (const r of rows) _configCache[r.config_key] = r.config_value;
  }
  return _configCache;
}
function invalidateSystemConfig() { _configCache = null; }
```

**变更点**:
1. recalcMainPlanDates: 3 行 cfgRaw+for → `const cfg = getSystemConfig();`
2. auto-schedule: 同上
3. PUT `/api/system-config/:key` UPDATE 后加 `invalidateSystemConfig();`
4. PUT `/api/config/system/:key` UPDATE 后加 `invalidateSystemConfig();`

**烟测**:
- POST `/main-plan` due=08-15 plan_qty=1000 → cutting=08-01 sewing=08-08~10 ✓ (缓存命中)
- PUT `shipping_buffer_days` 5→7 → POST `/main-plan` → sewingEnd = 08-15 - 7 = 08-08 ✓ (新值生效)
- PUT 还原 7→5 → POST 恢复 ✓

## 段10 — P3 CuttingDispatch progress tab 后端化 (1 项, commit 3d92b24)

**目标**: `completionRows.value = (data || []).filter(r => r.under_order === 1 || (r.first_actual + r.second_actual) > 0)` 改后端 SQL HAVING。

**改造**:
- server.js: `/api/cutting-completion` 加 `?mode=active` 模式,HAVING 子句复用 SELECT 中的 CASE 表达式
- CuttingDispatch.vue: `loadCompletion()` 传 `params: { mode: 'active' }`,去掉 .filter
- `checkUnderOrder()` 保留不带参数的全量查询(需找特定行)

**SQL 关键**: SQLite HAVING 不支持 SELECT 别名,需重写完整表达式:
```sql
HAVING (
  CASE WHEN scs.plan_qty > 0 AND
    COALESCE(SUM(CASE WHEN ap.schedule_type='cutting' AND COALESCE(ap.is_second_inspection,0)=0
      THEN ap.completed_qty ELSE 0 END), 0) < scs.plan_qty
    THEN 1 ELSE 0 END = 1
  OR COALESCE(SUM(CASE WHEN ap.schedule_type='cutting' AND COALESCE(ap.is_second_inspection,0)=0
       THEN ap.completed_qty ELSE 0 END), 0)
   + COALESCE(SUM(CASE WHEN ap.schedule_type='cutting' AND COALESCE(ap.is_second_inspection,0)=1
       THEN ap.completed_qty ELSE 0 END), 0) > 0
)
```

**烟测**:
- 反向验证:`HAVING 1=0` → 0 行,`HAVING 1=1` → 2325 行,`HAVING active` → 2325 行 (当前数据 100% 满足 active 条件,数据特性)
- 等价性:后端 2325 = 前端 filter 应得 2325 ✓

**为什么 2325=all**: 当前 style_color_size 数据全部 plan_qty>0 且 100% under_order=1 (first_actual<order_qty),所以 HAVING 实际不过滤。SQL 语法和逻辑正确,数据特性导致命中 100%。
